# CSDN 风格博客 UI 改造技术实施方案

## 1. 文档目的

本文档用于指导 `zhuwei-blog` 从当前的深色技术博客视觉，改造成参考图所体现的浅色内容平台风格。

参考图只用于提取布局、信息层级和视觉语言，不复制 CSDN 的品牌标识、文案和业务功能。项目中不存在或当前没有后端能力支持的登录、关注、AI 搜索、创作中心等功能不纳入本次改造。

本方案覆盖：

- 公共导航与主题系统
- 首页分类侧栏、作者信息和文章列表
- 博客归档页与标签页
- 文章详情页与文章目录
- 移动端响应式
- 现有阅读、评论、点赞数据的兼容
- 可访问性、SEO、性能和测试验收

## 2. 当前项目基线

### 2.1 技术栈

- 框架：Astro 6
- 内容来源：Astro Content Collections
- 部署平台：Cloudflare Pages
- 动态数据：Cloudflare Functions + D1
- 样式方案：全局 CSS，没有引入 UI 框架
- 主题能力：亮色、暗色、跟随系统三种模式
- 文章能力：Markdown、目录、阅读量、点赞、评论、RSS、SEO

### 2.2 当前页面结构

- `/`：首页，包含大尺寸 Hero、分类标签、统计栏、置顶文章、文章列表和热门文章侧栏
- `/blog/`：博客归档页
- `/tags/[tag]/`：标签和分类聚合页
- `/blog/[slug]/`：文章详情页
- `/about/`：关于页面

### 2.3 当前实现中的主要问题

1. `BaseLayout.astro` 同时加载 `base.css`、`home.css` 和 `detail.css`，所有页面都会下载并应用不需要的页面样式，增加样式冲突风险。
2. 首页视觉以深色、黄色、方角和大 Hero 为主，与参考图的浅色、高信息密度内容平台风格差异较大。
3. 首页分类入口由 `tabItems` 硬编码，分类数量没有从文章集合动态计算。
4. 首页文章列表和 `PostCard.astro` 存在两套相似结构，后续维护容易出现不一致。
5. 首页、归档页、标签页分别包含相似的动态指标请求脚本，存在重复代码。
6. 全局正文字体将 `JetBrains Mono` 放在首位，不适合中文内容平台的长文本阅读。
7. 当前搜索区域只是视觉元素，没有完整搜索行为。
8. `home-premium.css`、`detail-premium.css`、`minimal-site.css` 等历史样式没有参与正式路由，本次不直接删除，避免误伤历史参考稿。

## 3. 改造目标

### 3.1 视觉目标

- 默认呈现浅色内容平台风格。
- 使用白色内容面板、浅灰页面背景、细分割线和弱阴影。
- 使用橙红色作为品牌强调色，替代当前大面积黄色。
- 提高文章列表的信息密度，减少大面积装饰性区域。
- 保持标题、摘要、元数据之间清晰的阅读层级。
- 不使用 CSDN Logo、吉祥物、品牌名称和受保护素材。

### 3.2 功能目标

- 保留现有文章路由、Markdown 内容和 SEO 数据。
- 保留 D1 阅读量、评论数和点赞能力。
- 分类数量根据文章数据自动生成。
- 首页、归档页、标签页复用同一套文章列表组件。
- 搜索采用纯前端静态过滤，不新增服务端搜索接口。
- 主题切换继续可用，并兼容用户已保存的主题选择。

### 3.3 非目标

本次不实现：

- 用户注册和登录
- 关注作者
- 第三方账号绑定
- AI 搜索
- 创作中心和在线编辑器
- 站内消息中心
- 文章付费、会员和积分体系
- 新的评论后端或数据库表
- 分页接口
- CSDN 品牌资源和一比一业务功能复制

### 5.2 字体

全局正文建议：

```css
font-family: "Noto Sans SC", -apple-system, BlinkMacSystemFont,
  "Segoe UI", "Microsoft YaHei", sans-serif;
```

代码块、行内代码和快捷键继续使用：

```css
font-family: "JetBrains Mono", "SFMono-Regular", Consolas, monospace;
```

### 5.3 尺寸规范

| 元素 | 桌面端建议值 |
| --- | --- |
| 顶部导航高度 | 64px |
| 页面最大宽度 | 1440px |
| 左侧分类栏 | 236px |
| 主布局间距 | 16px |
| 作者概览内边距 | 24px 32px |
| 文章列表项内边距 | 24px 32px |
| 文章标题 | 20px / 1.45 |
| 文章摘要 | 14px / 1.75 |
| 元数据 | 13px |
| 正文最大可读宽度 | 820px 左右 |
| 文章目录宽度 | 280px |

### 5.4 交互规范

- 普通动画时长统一为 `150ms`。
- Hover 只改变文字色、背景色或边框色，不使用明显位移。
- 所有可点击控件必须有明确的 `:focus-visible` 状态。
- 移动端可点击区域高度不低于 40px，关键按钮不低于 44px。
- 支持 `prefers-reduced-motion`，用户关闭动画时禁用平滑过渡。

## 6. 数据层设计

### 6.1 站点配置集中化

新增 `src/config/site.ts`，集中维护站点和作者信息，避免 Header、Footer、SEO 和作者概览重复硬编码。

建议字段：

```ts
export const SITE_CONFIG = {
  name: "ZHUWEI.BLOG",
  author: "ZhuWei",
  description: "专注于后端工程、网络协议与 AI 自动化",
  role: "后端工程 / AI 自动化实践",
  avatarText: "Z",
  github: "",
  rss: "/rss.xml"
};
```

未配置的外部链接不渲染入口，不创建无效的 `href="#"`。

### 6.2 分类统计

在 `src/lib/blog.ts` 新增分类聚合函数，由所有列表页共用。

返回数据结构：

```ts
interface CategoryStat {
  name: string;
  slug: string;
  count: number;
  color: ColorName;
}
```

处理规则：

1. 只统计非草稿文章。
2. 以文章 `category` 字段分组。
3. `slug` 继续复用现有 `slugifyTag()`。
4. 默认按文章数量降序，同数量按分类名排序。
5. 分类颜色取该分类第一篇文章的 `color`，没有值时回退为 `blue`。
6. 列表第一项固定为“全部文章”，数量为文章总数。

关键聚合逻辑、排序原因和回退策略需要添加中文注释。

### 6.3 作者统计

作者概览使用现有数据计算：

- 文章数：`posts.length`
- 总阅读量：优先使用接口返回的动态指标之和，接口失败时使用 frontmatter 初始值
- 分类数：去重后的分类数量
- 总字数：按现有文章正文统计逻辑计算

不展示没有真实来源的关注数、粉丝数和虚构访客数。

### 6.4 动态指标统一

新增 `public/scripts/post-metrics.js`，替代首页、归档页和标签页中重复的指标请求代码。

处理流程：

1. 查询页面上的 `[data-post-slug]`。
2. 去重并批量请求 `/api/posts/metrics?slugs=...`。
3. 更新对应文章的阅读量和评论数。
4. 重新计算页面总阅读量。
5. 请求失败时保留构建期 frontmatter 数值，不显示错误占位。
6. 更新 DOM 时使用 `textContent`，不使用 `innerHTML`。

现有 Cloudflare Functions 和 D1 数据结构无需修改。

## 7. 组件设计

### 7.1 `Header.astro`

职责：

- 渲染品牌 Logo
- 渲染主要导航
- 渲染站内搜索表单
- 渲染 RSS、主题切换和关于入口
- 标记当前路由

具体调整：

- 高度改为 64px，白色背景，底部细边框。
- Logo 保留 `ZHUWEI.BLOG`，使用橙红色强调 `.BLOG`。
- 桌面端搜索框居中或靠右，占用 280-360px。
- 搜索使用真实 `<form>` 和 `<input name="q">`，提交到 `/blog/`。
- 主题按钮保留现有 `data-theme-toggle` 行为。
- 不增加登录、创作、消息等无功能按钮。
- 窄屏隐藏次要导航，通过简化导航而不是增加复杂抽屉菜单完成第一阶段适配。

### 7.2 新增 `CategorySidebar.astro`

输入：

```ts
interface Props {
  categories: CategoryStat[];
  activeCategory?: string;
  hotPosts?: CollectionEntry<"blog">[];
}
```

职责：

- 渲染“分类专栏”标题。
- 渲染全部文章及各分类数量。
- 使用分类首字或简洁内联 SVG 作为图标，不依赖外部图片。
- 当前分类通过 `aria-current="page"` 和品牌色标识。
- 桌面端支持 `position: sticky`，顶部偏移为导航高度加 16px。
- 可选渲染热门文章，复用现有真实阅读量。

### 7.3 新增 `AuthorOverview.astro`

输入：作者配置、文章数、分类数、总阅读量、总字数。

职责：

- 渲染头像或文字头像。
- 渲染作者名、定位和博客简介。
- 渲染真实统计数据。
- 渲染存在且有效的外部链接。
- 不渲染“关注”按钮，因为项目没有关注能力。

### 7.4 重构 `PostCard.astro`

`PostCard.astro` 作为首页、归档页、标签页唯一的标准文章列表项。

建议 Props：

```ts
interface Props {
  post: CollectionEntry<"blog">;
  pinned?: boolean;
  isNew?: boolean;
  showCategory?: boolean;
}
```

DOM 层级：

```text
article.post-list-item
├── 标题行
│   ├── 置顶 / 最新标签
│   └── 标题链接
├── 摘要
└── 元数据
    ├── 分类
    ├── 发布时间
    ├── 阅读量
    ├── 评论数
    └── 阅读时长
```

实现要求：

- 外层使用 `<article>`，标题链接单独使用 `<a>`，避免整个卡片都是链接导致内部无法扩展其他操作。
- 标题最多两行，摘要桌面端最多两行。
- 置顶文章使用“置顶”标签，不再使用大尺寸封面。
- 保留 `data-post-slug`、`data-post-views` 和 `data-post-comments`。
- 不依赖文章封面字段，因为当前内容 schema 没有统一封面。

### 7.5 新增 `PostList.astro`

职责：

- 接收文章数组并循环渲染 `PostCard`。
- 统一处理第一篇文章的“最新”标识。
- 统一处理置顶文章顺序。
- 空列表时渲染标准空状态。

这样首页、归档和标签页只负责准备数据，不再复制文章循环结构。

### 7.6 `Sidebar.astro` 与 `Toc.astro`

- `Sidebar.astro` 继续服务文章详情页。
- 文章目录卡片改为白色轻边框样式。
- 桌面端目录吸顶，最大高度为视口减去顶部导航和安全间距。
- 目录过长时内部滚动。
- 保留现有目录激活脚本，不改 heading slug 规则。

### 7.7 `Footer.astro`

- 改为浅灰背景或白色顶边框。
- 只保留真实可访问链接。
- 减少垂直高度，避免列表页底部出现大面积空白。
- 暗色主题通过变量自动适配。

## 8. 页面级实施方案

### 8.1 首页 `/`

调整 `src/pages/index.astro`：

1. 删除当前大尺寸 `.hp-hero`。
2. 删除硬编码的 `tabItems`。
3. 调用分类聚合函数生成左侧分类数据。
4. 计算作者真实统计数据。
5. 页面主体改为 `CategorySidebar + 主内容区` 两栏。
6. 主内容顶部渲染 `AuthorOverview`。
7. 使用 `PostList` 渲染全部文章。
8. 置顶文章排在第一位并显示“置顶”标签，避免与普通文章重复。
9. 引入统一的动态指标脚本。

### 8.2 归档页 `/blog/`

调整 `src/pages/blog/index.astro`：

- 使用与首页一致的页面骨架。
- 左侧显示分类专栏。
- 主区域顶部显示“博客归档”和文章总数，不重复作者横幅。
- 支持读取 URL 查询参数 `q` 并在客户端过滤标题、摘要、分类和标签。
- 搜索前所有文章仍输出在静态 HTML 中，保证 SEO 和无 JavaScript 降级。
- 删除页面内重复的指标请求脚本。

### 8.3 标签页 `/tags/[tag]/`

调整 `src/pages/tags/[tag].astro`：

- 使用相同分类侧栏和文章列表。
- 当前分类或标签高亮。
- 顶部显示当前标签名、文章数和简短说明。
- 不再额外渲染大尺寸 FeaturedPost。
- Hermes Agent 的专属置顶头图不再作为列表页核心展示；原图片保留给文章正文或 Open Graph 使用。

### 8.4 文章详情页 `/blog/[slug]/`

调整 `src/layouts/PostLayout.astro` 和 `src/styles/detail.css`：

- 页面背景统一为浅灰色。
- 文章容器使用白色面板。
- 标题、摘要、分类和元数据重新排列，减少顶部空白。
- 正文宽度控制在适合阅读的范围。
- 右侧目录吸顶。
- 相关文章改成紧凑文本列表。
- 点赞和评论逻辑不变，只更新视觉。
- Markdown 表格、代码块、引用块和提示块需要同时验证亮暗主题。

### 8.5 关于页 `/about/`

- 复用作者配置。
- 使用与文章详情相同的白色内容面板。
- 保留现有内容，只统一排版和信息卡片风格。

## 9. CSS 组织调整

### 9.1 样式职责

建议最终职责：

- `base.css`：Reset、设计变量、字体、Header、Footer、主题、通用按钮和全局响应式规则
- `home.css`：首页、归档页、标签页共用的列表布局、分类侧栏、作者概览和文章列表
- `detail.css`：文章详情、Markdown 正文、目录、互动区和评论区

### 9.2 按页面加载 CSS

调整 `BaseLayout.astro`：

- 只导入 `base.css`。
- 首页、归档页和标签页分别导入 `home.css`。
- `PostLayout.astro` 导入 `detail.css`。
- `about.astro` 根据实际使用的组件导入 `detail.css`。

这样可以减少页面无关 CSS，并降低同名选择器覆盖风险。

### 9.3 命名策略

- 公共组件使用明确前缀，例如 `.site-*`、`.post-*`、`.category-*`。
- 详情页继续使用 `.article-*`。
- 逐步替换当前 `.hp-*` 命名，不在一次提交中保留两套长期兼容层。
- 禁止新增无语义的 `.left`、`.box2`、`.red-text` 等选择器。
- CSS 大段区域使用中文注释标明职责，例如“顶部导航”“分类侧栏”“文章列表”“移动端适配”。

## 10. 搜索实施方案

本次实现轻量静态搜索，不增加后端接口。

### 10.1 提交流程

1. Header 搜索表单提交到 `/blog/?q=关键词`。
2. 归档页静态输出全部文章。
3. 客户端脚本读取 `URLSearchParams` 中的 `q`。
4. 根据文章 DOM 上的标题、摘要、分类和标签数据进行过滤。
5. 显示“找到 N 篇相关文章”或标准空状态。

### 10.2 降级策略

- JavaScript 失败时仍显示完整归档列表。
- 搜索不影响文章页面的静态生成和索引。
- 不做模糊分词、高亮、联想词和跨正文全文搜索。

## 11. 主题系统方案

### 11.1 默认主题

- 新访客默认使用亮色，以保证与目标设计一致。
- 已经保存 `light`、`dark` 或 `system` 的用户继续使用原设置。
- 主题切换顺序建议为：亮色 -> 暗色 -> 跟随系统。

### 11.2 首屏闪烁控制

保留 `BaseLayout.astro` `<head>` 中的内联主题初始化脚本，在 CSS 加载前写入 `data-theme`，避免页面先亮后暗或先暗后亮。

### 11.3 浏览器颜色

## 13. SEO、语义和可访问性

### 13.1 SEO

- 保留现有 canonical、Open Graph、RSS 和 JSON-LD。

- 页面主体使用唯一 `<main>`。
- 分类区域使用 `<aside>` 和带标题的 `<nav>`。

- 所有链接和按钮支持 Tab 导航。
- Focus 样式不能只依赖颜色变化。

## 14. 性能约束

- 不引入大型 UI 组件库。
- 不引入新的客户端框架或 hydration island。
- 分类图标优先使用内联 SVG 或文字标识。
- 列表页不加载无用大图。
- CSS 通过页面级导入减少无关资源。
- 动态指标继续使用单次批量请求，禁止每篇文章单独请求。
- 字体加载失败时使用系统中文字体，不阻塞内容展示。

## 18. 风险与处理策略

### 18.1 全局 CSS 冲突

风险：当前三个主要 CSS 文件由 BaseLayout 全量加载，重构过程中容易发生覆盖顺序问题。

处理：第一阶段先完成按页面加载 CSS，再开始大规模替换选择器；每完成一个页面就删除对应旧规则，不保留长期双轨样式。

### 18.2 动态指标与静态初始值不一致

风险：页面首屏先显示 frontmatter 数值，接口返回后发生变化。

处理：保留静态值作为降级，动态更新只替换文本，不改变元素尺寸；数字使用 tabular numbers 降低布局抖动。

### 18.3 分类颜色不一致

风险：同一分类下文章可能配置不同 `color`。

处理：分类侧栏使用稳定的分类颜色映射或首篇文章颜色，不逐篇变化。后续如需严格管理，再扩展独立分类配置，不在本次修改 content schema。

### 18.4 搜索能力有限

风险：纯前端搜索不能搜索完整正文，也不支持中文分词。

处理：明确搜索范围为标题、摘要、分类和标签。文章数量明显增长后，再评估 Pagefind 或服务端搜索。

### 18.5 参考图业务元素缺失

风险：完全照搬会出现无功能按钮和虚假数据。

处理：只复用视觉层级和布局方法；不存在的业务入口不渲染，所有统计必须来自项目真实数据。
